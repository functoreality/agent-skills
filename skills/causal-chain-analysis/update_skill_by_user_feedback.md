任务：总结本次 skill 执行的经验和教训，并将一般化的原则更新到 skill 当中，使得日后使用它的 AI Agent 的分析结果质量能有明显的提升。

建议的分析模式：
1. 思路归因：对于每一个做得好或者有问题的点，不能只停留在结果的形式上，还应进一步挖掘它底层的思维模式，就是通过什么样的分析方法和逻辑推演过程，可以达到这样的结果。
2. 泛化表述：在分析出其中的思维方式以后，尝试将它重新表述，组织整理成可以泛化的形式，使得它对其他的问题同样也能够适用，而不是过拟合到当前这样一个特殊的问题场景上。
	* 这个 skill 未来可能用到很多种不同的场景，包括分析物理、机械问题，分析 AI 训练、分析团队组织建设、分析目标任务管理，分析机制设计等等。
	* 如认为有必要查看具体问题案例集合（不总是如此），可见 `~/Documents/projects/agent-skills/test_cases/causal-chain-analysis/README.md`
3. 设计改动：设计具体更新方案，skill 哪些地方需要修改，更新内容的文字如何表述。这样的更新应反映尽量通用的思维方式，并且文字表述尽量凝练、言简意赅。
4. 推演验证：为验证更新方案反映的是有效、可复用的思维方式，而非仅仅是当前案例的浮浅形式描述，应设想如果使用更新后的 skill 进行问题分析，是否确实能达到理想的效果？
	* 保有效性：设想另一个 AI 使用更新后的 skill 执行本次相同的分析步骤，推演它是否可以复现本次的成功经验，或不再出现本次的问题。
	* 防副作用：设想另一个 AI 使用更新后的 skill 执行本次分析过程中的其他步骤，或者其他问题（可能来自不同领域）可能涉及的某个分析步骤，推演它是否可能被其误导而跑偏，而使用原版的 skill 时则可以正常进行。如果是，当前的 skill 更新方案应重新设计。
	* 比多版本：如果设想出了多种不同的 skill 更新方案（有时也应当如此），对这些更新方案分别推演，验证它们的有效性和副作用，然后从中选出表现更好的方案。这个最优方案后续也可以进一步优化调整。

要求：复盘分析逐步推进，步步为营。
* 如果某步结果拿不准，请停下来向用户确认，不要凭主观臆断直接进入下一步。
* 与用户讨论清楚更新方案后，才允许实际执行更新。
