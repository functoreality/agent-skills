# 因果链分析 Skill

## 概述

因果链分析是一种系统性的问题根因分析方法。它从用户提供的原始问题（表层缺点）出发，先分析本质上需要解决的初始缺点，并通过反复追问"这个缺点如何产生"，逐层深入挖掘直接原因，最终建立起一条条从初始缺点到末端缺点的因果链条。

**核心输出**：一个Markdown文件，包含完整的因果链分析结果，使用缩进表示层级关系。

---

## 何时使用此Skill

当用户提到以下内容时，应调用此skill：
- "分析因果链"
- "帮我做因果链分析"
- "进行因果链分析"

如果用户只是对某种情况感到不满意，希望能解决眼前的问题，却想不出好的解决办法，则：
1. 先自行评估，如果对这个问题的产生机制（在理论层面）做一次完整充分的分析，是否有助于产生更多的候选方案
2. 如果是，询问用户：是否愿意由 AI 助手你来系统地梳理一遍问题出现的原因，这或许能帮助想出更多的解决思路
3. 用户同意后，调用此skill
4. 若 1，2 中任何一个的答案为否，可忽略此skill

## 分析流程

### 步骤1：理解用户问题

1. 仔细阅读用户描述的问题或缺点
2. 识别用户提供的初始"表层缺点"
3. **关键**：分析这个表层缺点，判断它是否是"本质初始缺点"，还是用户最直接看到的表层缺点

### 步骤2：推导本质初始缺点

如果用户提供的只是一个导致不满的直接现象，而非根本上造成不满的初始缺点，需要进行推导：

1. 询问"这个表层缺点会导致什么？"
2. 推演它的结果
3. 继续追问，直到找到用户根本上想解决的问题——本质性的初始缺点
4. 如果你认为这样的初始缺点不止一个，可以分别展开因果链分析，但不宜过多
5. 如果你识别出了 3 个或以上的初始缺点，请向用户确认，避免出现理解偏差

**示例**：
- 用户说"冬天开门被静电打到"——这需要推导为"电流刺激神经末梢导致疼痛"，而"疼痛"才是真正让用户不满的初始缺点——因为如果换用神经不敏感的手背，即使被电到，也没有明显感觉
- 用户说"感到疼痛"——这已经是初始缺点，因为尽管它会进一步导致"心情不好"，但用其他办法来解决已经意义不大
- 用户说"油漆从油漆箱中溢出"——这可能是初始缺点，也可能不是，因为它确实会导致"油漆浪费"、"地面污染"，但这些可能已经超出了当前项目所需要关注的范围。遇到这种情况，可以考虑向用户确认

**重要**：如果对初始缺点的理解有疑问，**先向用户确认**再继续分析。

确认时需要留意，用户可能对"初始缺点"这个概念不够了解。可以考虑换成更通俗的表述，比如"本质上想解决的问题"。

### 步骤3：构建因果链

对于每一个发现的缺点，按照以下规则逐层寻找下一级直接原因：

1. **寻找直接原因**：只找物理上直接接触的组件所引起的原因，而非间接原因
2. **善用科学公式**：比如，如果本层次的缺点是"摩擦力大"，利用"摩擦力=压力×摩擦系数"，就应该能够在压力和摩擦系数这两个方面找到下一层级的缺点
3. **默认And关系**：同一层级下的多个原因，默认是And关系（共同作用造成）
4. **标记其他关系**：如果某原因是Or关系（任一即可造成），在展开前的上一个缩进层级的最后标记 `（or）`。加法、乘法关系相应地标记 `（+）`，`（×）`。
5. **追问下层原因**：对每个中间缺点重复"它为什么发生"，直到达到末端
6. **只管客观因果**：所有因果关系仅限于客观规律（为何发生），不讨论主观动机（为何这么做）。诸如"需要xx"、"为了xx"的因素不是"缺点"，不能作为下一层的原因
7. **规范缺点表述**：每个缺点应采用"主语+谓语"的基本句式，确保表述清晰准确，是可检验的客观事实。

**And/Or/+/× 关系表示规范**：

```markdown
# 示例1：A的原因是B和C（And关系）
* 物体着火
  * 可燃物存在
  * 氧气存在
  * 温度达到燃点

# 示例2：A的原因是B或C（Or关系）
* 喝的水受污染（or）
  * 水源污染
  * 管道污染
  * 容器污染

# 示例3：A的原因是B+C（+关系）
* 碳排放过多（+）
  * 电力热能碳排放多
  * 工业碳排放多
  * 交通碳排放多

# 示例4：复杂情况 - A的原因是B × C，而B的条件是D and E
* A（×）
  * B
    * D
    * E
  * C
```

**重复因素表示规范**：

```markdown
# 示例：D 同时作为 B、C 的原因，只在第一次出现时展开
* A
  * B
    * D
      * F
      * G
    * E
  * C
    * D（上方已分析）
    * H
```

互相促进的因素（正反馈回路）与之类似：

```markdown
# 示例：B 促进 A，A 反过来也会促进 B
* A（+）
  * B（×）
    * A（循环因果）
    * D
  * C
# A → B → C → A 及更复杂的情形同理
```

### 步骤4：确定末端缺点

当满足以下任一条件时，停止追问，该点即为末端缺点：

1. 达到物理、化学、生物或几何等领域的极限
2. 达到自然现象（如惯性、引力、人体生理机制等，它们是导致问题的缺点，但无法干预改变）
3. 达到系统设计决策（如"油漆箱容量太小"，不必分析为何这么设计；继续分析"长/宽/高/形状"则是可以的）
4. 达到法规、国家或行业标准限制
5. 无法继续找到下一层原因
6. 达到成本极限或人的本性
7. 继续分析将与本项目无关

### 步骤5：因果链结构检查（非常重要！）

**在结束分析之前，必须先进行因果链结构检查。** 这一步至关重要，需要反复检查多遍，直到确认因果链结构完整正确。

将因果链分析写入文件后，暂停一下，每轮进行以下 6 项检查，重复多轮直到改不动为止：

**重要**：对每一项检查，如果发现错误就立即修改文件，不能等完成所有 6 项检查后才一起修改。

#### 检查一：每个缺点因素的表述是否满足要求——主语+谓语，准确具体，原子化，客观事实？

因果链分析中的每个缺点因素必须是**主语+谓语形式的，准确具体的，原子化的客观事实**。
- **主语+谓语形式**：每个因素应采用"主语 + 谓语"的基本句式，表述一个清晰、明确、可检验的事实。
- **准确具体**：每个主语、谓语都越具体、越准确越好，不允许模糊泛指
- **原子化**：每个因素应是不可再分的单一事实，不包含隐藏的因果关系或并列关系。
- **客观事实**：每个因素必须是可观测、可检验的描述，不依赖于纯粹价值判断或主观动机。

**表述规范**：

每个缺点因素必须符合以下要求：

1. **主语准确具体**
  - 主语应指代具体的实体、属性或度量，避免模糊泛指。
  - **改进示例**：将`“输入句子长度”大`改为`“输入句子字数”多`；将`“系统响应”慢`改为`“系统响应时间”长`。
  - **精确区分**：务必区分非常相似、但实际上不同的概念。比如，`参与注意力计算的元素个数`、`输入数据总token数`、`输入文本单词数/图像像素个数/视频帧数`，是三个完全不同的东西，绝不能用笼统的`序列长度`糊弄过去。
  - **确保专业**：因果链分析对表述精确性的要求相当高。特定领域中经常使用的表述，在因果链分析中可能完全不合适。

2. **谓语清晰准确**
  - 谓语可以是形容词（描述状态/性质）或动词（描述动作），可以有宾语；如果涉及比较，可以有比较的对象。
  - 谓语应具体、可检验，避免抽象的主观评价。
  - 允许使用"太多"、"效率低"等定性描述，因为它们指向可观测、可比较的属性，能**提示明确的改进方向**。
  - **禁止使用**："不合理"、"设计不良"等纯价值判断、缺乏明确比较基准的描述。这类表述过于抽象模糊，无法指导具体改进。

3. **保持原子性**
  - 一个因素只表述一个事实，不隐含因果关系。
    - **错误示例**：`水温高导致气体溶解度低`（包含"导致"关系，应拆分为两个因素：`气体溶解度低` → `水温高`）
  - 不合并多个独立因素。
    - **错误示例**：`电压存在 AND 物体导电`（应拆分为两个同级因素：`电压存在`和`物体导电`）

4. **避免主观动机**
  - 不使用"需要xx"、"为了xx"、"旨在xx"等表述动机或目的的短语。
  - **错误示例**：`需要提高系统吞吐量`（应改为客观事实：`系统吞吐量低`）
  - **错误示例**：`为了减少延迟而使用了缓存`（只能说客观缺点：`缓存一致性开销大`）

5. **必要时补充状语**
  - 在少数有必要的情况下，可使用括号补充时间状语、条件状语等背景信息。
  - **示例**：`（高峰期）CPU使用率过高` 或 `（打印头回抽时）材料中断`
  - **注意**：不能讨论原因和结果，这些应独立成点以满足原子性。

**检查方法**：
1. 逐行检查每个缺点因素，判断其是否符合主语+谓语形式。
2. 查找以下问题迹象：
  - 包含"导致"、"引起"、"使"等表示因果关系的词汇
  - 包含"和"、"与"、"及"、"或"、"且"等连接多个因素的词汇
  - 包含"要"、"为"、"旨在"、"以"等表示动机的词汇
  - 包含"不合理"、"不良"、"糟糕"等价值判断词汇
  - **关键建议**：使用 grep 等工具在分析结果文档中搜索相关关键词
    - 注意，你仍然需要亲自进行基于语义的检查，因为关键词列表无法覆盖所有情况
3. 对于不符合要求的因素，按上述规范进行拆分、改写或删除。

**示例对比**：

| 不符合要求的表述 | 符合要求的表述 |
|----------------|----------------|
| `内存访问不合理` | `内存访问频率过高` |
| `序列长度大` | `参与注意力计算的元素数量多` |
| `系统吞吐量需要提高` | `系统吞吐量低` |
| `水温高导致气体溶解度低` | `气体溶解度低`（原因：`水温高`作为下层因素） |
| `电压存在且物体导电` | `电压存在`<br>`物体导电`（作为两个同级因素） |

#### 检查二：是否有跳步？

检查是否出现"逻辑跳跃"——即原来分析出A的原因是B，但转念一想，其实A的直接原因应该是C，C的直接原因才是B。
比如，对于"（端起热水杯子时）手感觉烫"，不能想当然地说"杯子中水的温度太高"，因为它不是造成缺点的直接原因。应该分解成：手感觉烫 → 手部温度感觉神经受刺激 → 手表面温度太高。而"杯子中的水温度太高"，属于更底层的缺点。

**错误示例**：
```markdown
* A
  * B
```
如果B不是A的直接原因，A的直接原因应该是C，而C的原因是B，则应该修正为：
```markdown
* A
  * C
    * B
```

#### 检查三：是否有遗漏因素？（勇敢想象，多写公式）

检查是否有遗漏的其他因素。比如之前识别出A的直接原因是B and C，但现在想想发现其实应该是B and C and D，而D这个因素是之前没有想到的——即使B、C都成立，如果D不满足，A也不会发生。

* 勇敢想象：在B、C都发生的前提下，我就是希望阻止A发生，到底有没有办法？反复问自己这个问题。
* 多用公式：如果当时展开A没有使用科学公式，现在能不能用公式把A彻底拆解一遍？公式是避免遗漏的强大手段。
* 直观公式：如果没有"科学"公式，考虑能不能用一个可能不那么严谨、更多在抽象直觉层面上建立的公式/关系，用它来拆解一下A。

如果有遗漏，补充到相应的层级下。

#### 检查四：是否提前终止？

检查某个因素节点在之前的分析中是否过早终止——缩进层级到此为止，但其实现在看看它好像并不是末端缺点，可以继续展开出下一层的原因。

**检查方法**：对于每个叶子节点，问自己"这个原因还能继续追问'它发生的机制是什么'吗？"

注意别跑偏——追问涉及的所有因果关系仅限于客观规律（为什么发生），不讨论主观动机（为什么这么做）。不要分析系统为什么这么设计，参数为什么这么选取。如果出现了这样的分析，请把它删除。

#### 检查五：你能想到的所有解决方案，是否都对应末端缺点？

* 先别管你做过的分析，直接看用户的原始问题，或者识别出来的初始缺点，设想一些可能的解决方案
* 再回头检查你的分析：对每个这样的解决方案，检查它为何有效，是否是因为它解决了至少一个末端缺点
* 如果有它所解决的中间缺点，但没有直接对应的末端缺点，检查这是否意味着你的分析出现了：
  * 跳步，展开的原因不是直接原因
  * 某一层展开有遗漏因素
  * 有些因素不合适，应该删掉
  * 有的中间缺点实际上应该是末端缺点，之前错误地展开了设计动机，应当删除
  * 其他不合理之处
* 如果你认为它确实不应该对应任何末端缺点，需要有明确的理由

#### 检查六：是否有其他你觉得不合理之处？

一切你觉得不合理的地方都算：前后矛盾，不构成原因，等等等等

#### 检查流程

1. **第一轮检查**：完成初步因果链后，逐个检查每个因果关系
2. **如有错误**：立即编辑文件进行修正（不要等到 6 项检查都完成了再改）
3. **重复检查**：修正后再次检查，直到**连续两轮**检查都没有发现需要修改的地方
4. **走到下一步**：确认没有可以再更改的地方后，才开始启动子 agent 进一步检查

### 步骤6：子 Agent 改进审查

**开始前提**：如果你没有完成上一步的 6 项检查，并且确保连续两轮（相当于 2×6 项）这样的检查都没发现问题，不要进入这一步。

**执行条件**：如果你负责的是从头执行一次完整的因果链分析（而不是改进一个已有的分析结果文档），并且没有被告知"不要创建子 agent"，则执行以下操作：

1. 创建一个子 agent，让它：
  - 加载同一个"因果链分析"的 skill；如果可行，尽量提供完整的 `SKILL.md` 文件路径。
  - 让它改进你生成的因果链分析结果文档。请告知这个文档的完整路径。
  - 命令它**绝对不要再创建子 agent**。

**目的**：通过独立的、上下文无污染的子 agent 审查，发现主 agent 可能遗漏的问题，进一步提升因果链分析的质量。

2. 子 agent 改进结束后，根据它修改后的新版因果链分析文档，重新执行一轮步骤 5 所提到的 6 项检查，发现新的问题就接着修改。再也找不出更多问题以后，可以结束分析。

## 输出格式规范

### Markdown文件结构

```markdown
# 因果链分析报告

## 初始缺点
[本质初始缺点的描述]

## 因果链分析

[使用缩进表示层级的因果链]
[同级缺点的关系区分 And / Or / + / ×]
[Or关系在最后标记 `（or）`，+/× 同理，And 无需标记]

- 初始缺点
  - 直接原因1
    - 下一层原因
      - 下一层原因
  - 直接原因2（or）
    - 原因A
    - 原因B
  - 直接原因3
    - ...
```

### 缩进表示规则

- **使用 `- ` 或 `* ` 开头**表示因果节点
- 每往下一层增加 **2个空格**的缩进
- **不需要编号**（如1.1.1.2），纯缩进即可
- **父子关系**：缩进在下一层的因素是上一层因素的原因
- **同级关系**：同一缩进层级的多个因素，默认是And关系（共同出现时使上一级因素出现）
- **其他关系**：如果是Or关系，在它们所导致的上一层因素后面标记 `（or）`，+/× 类似

**结构示例**：
```markdown
- 手指感到疼痛
  - 手指的神经末梢痛觉感受器丰富
  - 电流通过手指
    - 电压存在
      - 电荷存在
        - 电荷产生
          - 摩擦产生电荷
            - 物体发生摩擦
              - 两物体接触
              - 两物体相对运动
            - 材料性质容易摩擦起电
        - 电荷不能导出
          - 衣物不导电
            - 衣物材料不导电
            - 衣物干燥
          - 空气不导电
            - 空气干燥
    - 回路形成
      - 手触碰金属门把手
    - 手是导体
    - 门把手是导体
  - 神经信号传递到大脑
```

## 分析过程中的注意事项

### 勇敢深入分析

- 末端应该是**有意义的终点**，不要提前终止分析（但如果确实满足末端条件，则应该终止）
- 高手的分析经常可以达到 20+ 层，拆解出 500+ 个因素节点。尽管不是所有问题都确实能拆解出这么多的因素节点，但你应该有努力深入的极致追求
- 如果你觉得过多的缩进层级不便于编辑和阅读，可以把一个复杂的子树独立出来，并在原有的主要分析结果里提及。例如：

```markdown
* A
  * B（or）
    * D（单独深入分析）
    * E
  * C

# 单独分析：D
* D（×）
  * F
    * H
    * I
  * G
```

- 如果你觉得有必要，可以启动子 agent 对某个子因素再展开深入分析

### 验证逻辑

- 每找到一个因果关系问自己："这些原因真的保证会导致这个结果吗？有没有其他依赖的前提条件？"
- 检查是否有逻辑跳跃（跳过了中间层级的直接原因）

### 因果链检查（核心步骤！）

**必须严格执行**：
1. 每轮进行 6 项检查（表述不当、跳步、遗漏、提前终止、解决方案涵盖、其他检查）
2. 发现问题立即编辑文件修正
3. 反复检查，直到连续两轮检查都没有错误
4. 确认无误后再进行下一步

### 与用户互动

- **初始缺点不确定时**：先确认再分析
- **分析方向迷茫时**：可以简要说明你的思路，询问用户意见
- **用户有新信息时**：及时更新分析
- **检查过程中有疑问时**：也可以询问用户

### 使用网络搜索

- 使用条件：除非用户明确提及，否则一般不需要使用网络搜索，缺失的背景信息可以直接询问用户。如果你确实认为需要，请向用户确认。
- 使用时机：在独立完成第一版分析之后，才允许调用网络搜索。不允许一上来就搜索网络信息。
- 使用方式：务必注意，网上找到的原因分析通常极度不专业、不可靠。请仅仅用它来完善你的因果链分析，帮助发现更多之前没有意识到的因素。绝对不要被别人的结论主导你的分析过程。

## 写入文件的时机

1. **创建文件**：开始分析时立即创建一个空的Markdown文件
2. **迭代编辑**：随着分析的深入，不断往文件中添加内容
3. **检查修正**：完成因果链后，进行检查，发现问题立即编辑

**文件命名建议**：
- `因果链分析_问题名称.md`
- `causal_chain_analysis_xxx.md`

**位置**：由AI根据上下文自行决定，除非用户指定了特定位置。

## 案例参考

### 案例：静电问题

**用户输入**："冬天开门总是被静电打到"

**分析过程**：
1. 表层缺点：开门被电打到
2. 推导本质初始缺点：静电产生电流 → 电流刺激神经末梢 → 疼痛产生
3. 逐层分析：
  - 电流 = 电压 + 介质导电 + 闭合回路形成
  - 电压 = 电荷累积
  - 电荷累积 = 电荷持续产生 + 电荷不能导出
  - 等等...
4. **因果链结构检查**（必须执行）：
  - 检查是否有表述不当
  - 检查是否有跳步
  - 检查是否有遗漏
  - 检查是否提前终止
  - 检查是否能涵盖解决方案
  - 检查是否有其他不足
5. 外部审查：启动独立的子 agent 进一步检查，改进因果链分析结果

## 开始分析

当用户请求因果链分析时：

1. 首先理解用户的问题
2. 判断是否需要推导本质初始缺点（如需要，向用户确认推导结果）
3. 创建Markdown文件开始记录
4. 按照上述流程进行分析
5. 完成后进行因果链结构检查
6. 检查通过后启动子 agent 进一步检查改进
